# Bingo Game — Implementation Plan

## 1. Summary

Build a multiplayer Bingo game using SvelteKit (Cloudflare adapter), Tailwind CSS, Drizzle ORM + Neon Postgres, and SvelteKit Remote Functions with **polling** (Plan B from the original plan) for real-time state sync. Cloudflare Workers are serverless, so `query.live` with in-memory event emitters is not viable — polling at 1-2s intervals is the pragmatic choice for a turn-based game with ≤5 players.

---

## 2. Database Schema (Refined Drizzle TypeScript)

### Changes from the original plan
- **UUIDs** for all PKs (better for distributed systems like Cloudflare)
- **Separate `code` field** on rooms: short 6-char alphanumeric for sharing (distinct from the UUID `id`)
- **`host_id`** on rooms: tracks who created the room
- **Removed `connected`** from players: ephemeral state, not worth persisting — presence can be inferred from recent activity or left for future work
- **`round_number`** on rounds: explicit ordering across rounds in the same room
- **Indexes** on frequently queried columns (room_id FKs, room code lookup, round status)
- **Enum types** via pgEnum for status fields instead of raw text

### Schema file: `src/lib/server/db/schema.ts`

```typescript
import { pgTable, uuid, text, integer, boolean, timestamp, jsonb, index } from 'drizzle-orm/pg-core';
import { pgEnum } from 'drizzle-orm/pg-core';

// Enums
export const roomStatusEnum = pgEnum('room_status', ['waiting', 'in_progress', 'round_ended']);
export const roundStatusEnum = pgEnum('round_status', ['setup', 'active', 'finished']);

// --- rooms ---
export const rooms = pgTable('rooms', {
  id: uuid('id').primaryKey().defaultRandom(),
  code: text('code').notNull().unique(),            // 6-char shareable room code
  hostId: uuid('host_id').notNull(),                 // FK -> players.id (added after players defined)
  maxPlayers: integer('max_players').notNull().default(5),
  status: roomStatusEnum('status').notNull().default('waiting'),
  winWord: text('win_word').notNull().default('BINGO'),
  createdAt: timestamp('created_at').notNull().defaultNow(),
}, (table) => [
  index('rooms_code_idx').on(table.code),
]);

// --- players ---
export const players = pgTable('players', {
  id: uuid('id').primaryKey().defaultRandom(),
  roomId: uuid('room_id').notNull().references(() => rooms.id, { onDelete: 'cascade' }),
  displayName: text('display_name').notNull(),
  joinOrder: integer('join_order').notNull(),
  createdAt: timestamp('created_at').notNull().defaultNow(),
}, (table) => [
  index('players_room_id_idx').on(table.roomId),
]);

// Add FK for rooms.hostId now that players exists
// (Drizzle handles this via SQL migration, not inline — see migration section)

// --- rounds ---
export const rounds = pgTable('rounds', {
  id: uuid('id').primaryKey().defaultRandom(),
  roomId: uuid('room_id').notNull().references(() => rooms.id, { onDelete: 'cascade' }),
  roundNumber: integer('round_number').notNull().default(1),
  status: roundStatusEnum('status').notNull().default('setup'),
  currentTurnPlayerId: uuid('current_turn_player_id'),  // nullable — null during setup
  lastCalledNumber: integer('last_called_number'),
  turnStartedAt: timestamp('turn_started_at'),
  setupDeadline: timestamp('setup_deadline'),            // 30s from round creation
  createdAt: timestamp('created_at').notNull().defaultNow(),
}, (table) => [
  index('rounds_room_id_idx').on(table.roomId),
]);

// --- player_boards ---
export const playerBoards = pgTable('player_boards', {
  id: uuid('id').primaryKey().defaultRandom(),
  roundId: uuid('round_id').notNull().references(() => rounds.id, { onDelete: 'cascade' }),
  playerId: uuid('player_id').notNull().references(() => players.id, { onDelete: 'cascade' }),
  grid: jsonb('grid').notNull(),                      // number[][] — 5x5 arrangement
  marked: jsonb('marked').notNull().default('[]'),     // number[][] of [row,col] positions
  sweptLines: jsonb('swept_lines').notNull().default('[]'), // string[] — line identifiers like "row-0", "col-3", "diag-1"
  points: integer('points').notNull().default(0),
  confirmed: boolean('confirmed').notNull().default(false),
  confirmedAt: timestamp('confirmed_at'),
}, (table) => [
  index('player_boards_round_id_idx').on(table.roundId),
  index('player_boards_player_id_idx').on(table.playerId),
]);
```

### Migration note
The `rooms.hostId` FK to `players.id` creates a circular reference at table creation time. In the Drizzle migration, we add it as an `ALTER TABLE` after both tables exist, or use a raw SQL step in the migration file.

### Remote functions (server-side)

**`src/lib/server/game/room.remote.ts`**
| Function | Type | Args | Purpose |
|---|---|---|---|
| `createRoom` | command | `{ displayName, maxPlayers }` | Create room + first player, return room code |
| `joinRoom` | command | `{ roomCode, displayName }` | Join existing room, assign join_order |
| `getRoomState` | query | `{ roomCode }` | Full room snapshot: players, round, boards |

**`src/lib/server/game/board.remote.ts`**
| Function | Type | Args | Purpose |
|---|---|---|---|
| `updateGrid` | command | `{ roomId, playerId, grid }` | Save reordering during setup |
| `confirmBoard` | command | `{ roomId, playerId }` | Lock board, start round if all confirmed |
| `callNumber` | command | `{ roomId, playerId, number }` | Call a number (validates turn), advance turn after 2s |
| `sweepLine` | command | `{ roomId, playerId, lineIds }` | Sweep completed lines, award points |
| `callBingo` | command | `{ roomId, playerId }` | Win attempt — must have 5 points and be on their turn |

### Polling strategy

- Client polls `getRoomState(roomCode)` every **1.5 seconds** while in a room
- After the acting player's own command, they re-fetch immediately for instant feedback
- Server stores `turn_started_at` and `setup_deadline` timestamps — clients compute remaining time from these, no separate timer endpoint needed

### Key game logic (in `state.ts`)

**Line detection:**
- 12 possible lines per board: 5 rows + 5 cols + 2 diagonals
- Each line is identified by a string key: `"row-0"`, `"col-3"`, `"diag-main"`, `"diag-anti"`
- A line is "completable" when all 5 cells in it are marked
- Player can sweep any completable line they haven't already swept

**Turn flow:**
1. Player clicks a cell on their board → `callNumber` command
2. Server validates: is it their turn? Is number already called? Is number on their board at that cell position?
3. Server marks the number as called in the round, and marks that position on ALL player boards
4. Server sets `turn_started_at` to now + 2s (the pause)
5. Client polls, sees the 2s pause, waits, then turn advances to next player
6. Turn advancement: on the server side, after the 2s `turn_started_at` window expires, the next poll from any client triggers the turn to advance (check `turn_started_at + 2s <= now`)

**Winning:**
- Points are awarded via `sweepLine`, not automatically
- A player with 5 points can call `callBingo` on their turn
- First player to successfully call Bingo wins
- Game doesn't end until someone calls — other players at 5 points can still catch up

```js
import adapter from '@sveltejs/adapter-cloudflare';
import { sveltekit } from '@sveltejs/kit/vite';
import tailwindcss from '@tailwindcss/vite';

export default {
  kit: {
    adapter: adapter(),
    experimental: {
      remoteFunctions: true
    }
  },
  compilerOptions: {
    experimental: {
      async: true
    }
  }
};

We need to move the adapter + kit config to `svelte.config.js` and remove the duplicate from `vite.config.ts`, since that's where SvelteKit expects it. The remote functions opt-in flags must go in `svelte.config.js`.

### `vite.config.ts` (simplify)

Remove the `sveltekit()` plugin config from here — let `svelte.config.js` own the SvelteKit configuration. Keep only `tailwindcss()` and `test` config.

## 5. Implementation Order

### Phase 1: Project setup & schema
1. Create `svelte.config.js` with adapter-cloudflare + remote functions opt-in
2. Simplify `vite.config.ts` (remove duplicate sveltekit config)
3. Rewrite `src/lib/server/db/schema.ts` with the refined schema
4. Generate and run Drizzle migration (`drizzle-kit generate` + `drizzle-kit push`)
5. Create `src/lib/server/game/utils.ts` — room code generation (6-char alphanumeric), default grid (1-25 shuffled)

### Phase 2: Lobby & room management
6. Build home page (`routes/+page.svelte`) — create room form + join room form
7. Implement `room.remote.ts` — `createRoom`, `joinRoom`, `getRoomState`
8. Build room page (`routes/room/[code]/+page.svelte`) — waiting lobby with player list
9. Build `Lobby.svelte` and `PlayerList.svelte` components

### Phase 3: Setup phase
10. Build `Board.svelte` + `Cell.svelte` — 5x5 grid with drag-and-drop reordering (using native HTML5 drag API) and click-to-edit fallback
11. Build `SetupPhase.svelte` — board + 30s countdown timer + confirm button
12. Implement `board.remote.ts` — `updateGrid`, `confirmBoard`
13. Implement round creation logic: when all players join, create round with `setup_deadline = now + 30s`
14. Auto-confirm players who don't confirm before deadline

### Phase 4: Play phase
15. Build `TurnIndicator.svelte` — current player name + last called number
16. Implement `callNumber` command — mark number on all boards, advance turn after 2s
17. Implement `state.ts` — line detection, scoring logic
18. Implement `sweepLine` command — validate 5-in-a-row, award point
19. Wire up polling loop in room page — `setInterval` + immediate re-fetch after commands

### Phase 5: Winning & game end
20. Implement `callBingo` command — validate 5 points, on turn, declare winner
21. Build `WinOverlay.svelte` — celebration display
22. Round-end state: show winner, offer "Play Again" (creates new round in same room)

### Phase 6: Polish
23. Mobile-responsive layout
24. Error handling & edge cases (disconnect, invalid moves, room cleanup)
25. Visual polish — animations for marking cells, line sweeps, turn transitions
